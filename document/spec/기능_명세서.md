# 기능 명세서 — 청림그룹사운드 플랫폼

> 버전: v1 (2026-07-02) — 현재 mock 구현(`platform/src/components/*`) 동작을 실서비스 요구사항으로 승격
> 표기: **[MOCK]** = 현재 mock 코드의 실제 동작(그대로 유지할 규칙), **[신규]** = mock에 없어 백엔드 설계 시 새로 정의한 규칙
> [ERD_명세서.md](./ERD_명세서.md), [IA_명세서.md](./IA_명세서.md) 와 세트로 사용

## 1. 인증 / 가입

### 1.1 로그인
- Kakao OAuth 단일 로그인 수단(`.env.local`의 `KAKAO_CLIENT_ID/SECRET`). 이메일/비밀번호 로그인 없음.
- 로그인 성공 후 `users.auth_user_id`(또는 `kakao_id`) 매칭 결과에 따라 분기:
  - 매칭되는 `users` 레코드 있음 → `/home`
  - 없음 → `/join` (가입 방식 선택)

### 1.2 가입 방식 선택 (`/join`)
- **[MOCK]** 카카오 로그인 직후 이름 표시(`?name=` 쿼리 또는 `localStorage.kakaoName`) — **[신규]** 실제로는 세션에서 가져와야 함(쿼리 파라미터로 이름을 신뢰하는 것은 보안상 부적절, 프론트 표시용으로만 세션 조회 API 사용).
- 두 카드: "기존 부원이에요" → `/join/existing`, "처음 가입해요" → `/join/new`.

### 1.3 기존 부원 인증 (`/join/existing`)
- **1단계**: 이름 + 인증키(또는 이름+기수) 로 로스터 대조. **[MOCK]** 은 하드코딩된 `ROSTER` 딕셔너리 대조 — **[신규]** 실서비스는 CSV로 선등록된 `users` 테이블에서 `name`+`gen`(또는 관리자가 부여한 별도 인증키) 조회.
- 매칭 실패 시 재시도 허용, 매칭 성공 시 이름/기수는 잠금(수정 불가, 자물쇠 아이콘) — 아래 필드만 편집:
  - 아바타 소스(카카오/기본), 닉네임(≤20자), 자기소개(≤60자), 세션(다중 선택)+세션별 경력 연차, 선호 장르, 학과(로스터 값 프리필), 학번, 학년(재학/휴학/졸업 포함), 전화번호(형식 자동 포맷), 전화번호 비공개 토글.
- 필수값: 세션 ≥1개, 학과, 학번, 학년, 전화번호.
- 제출 시 `users.auth_user_id` 연결 + 위 필드 UPDATE, `status`는 기존 값 유지(신규 가입 절차 아님).

### 1.4 신규 지원 (`/join/new`)
- 폼 필수값: 이름, 세션(≥1, 세션별 경력 연차 포함), 학과, 학번(숫자), 학년(YEAR_LIST — 재학 연차 + 휴학/졸업 포함), 전화번호(자동 포맷), 자기소개(≤200자), 희망 면접 슬롯(면접 슬롯이 존재할 경우 최소 1개, 다중 선택 가능). 닉네임/선호 장르는 선택.
- 제출 시 `applications` 레코드 생성, `status='new'`.
- **상태 추적 화면(StatusView)**: `applications.status`를 4단계 스테퍼로 표시 — `new→doc`, `interview→interview`, `pass/fail→result`(대기 표시는 `await`). `status='new'`인 동안에는 희망 면접 슬롯을 재선택 가능.
- **[MOCK]** 데모용 "상태 강제 전환" 컨트롤이 존재하나 이는 프리뷰 전용 — **[신규]** 실서비스에는 포함하지 않는다(지원자 본인이 자기 심사 상태를 바꿀 수 있으면 안 됨).

### 1.5 지원자 → 회원 전환
- 운영진이 `applications.status`를 `pass`로 확정하는 순간, `users` 레코드가 새로 생성되며 `status='PROBATION'`으로 시작. **[신규]** 이 트랜잭션은 지원서의 이름/세션/학과/학번/학년/전화번호/자기소개를 그대로 이관한다.
- `fail` 확정 시 회원 전환 없음, `applications` 레코드는 이력으로 남는다(삭제 안 함).

## 2. 홈 (`/home`)

- **[MOCK]** 완전 읽기 전용 대시보드, 별도 권한 게이트 없음(로그인만 되어 있으면 렌더).
- 구성 요소:
  1. 상단 인사말(닉네임) + 통계 카드(전체 부원 수, 활성 팀 수) — **[MOCK]** "주간 합주 14회"는 하드코딩값, **[신규]** 이번 주 `bookings`+`booking_templates` 실제 COUNT로 대체.
  2. **지금/다음 합주** 스트립: 오늘 요일의 예약 중 현재 시각 포함 구간(`current`)과 그 다음 시작 구간(`next`).
  3. **이번 주 타임테이블 미리보기**(compact 그리드), 클릭 시 `/timetable` 이동.
  4. **공지 미리보기**: 최근 공지 5건(정렬 기준은 4.2 공지 목록 규칙과 동일하게 pinned 우선 + 최신순 — **[신규]**, mock은 단순 배열 순서였음). 행 클릭은 목록(`/notices`)으로만 이동 — **[신규]** 상세(`/notices/[id]`)로 바로 이동하도록 개선 권장.
  5. **팀 현황**: 활성 팀 중 최대 5개(이름/팀장/인원/현재 합주곡/모집 배지), "전체 보기" → `/teams`.

## 3. 부원

### 3.1 부원 목록 (`/members`)
- 필터: 세션(다중), 기수, 운영진만, 화이트리스트만. 검색: 이름/닉네임/학과/세션 부분일치.
- 개인정보 표시는 `privacy_settings` 기준 마스킹(3.4 참고) — 목록 카드에는 마스킹 적용 후 값만 내려준다(서버에서 마스킹, 클라이언트 신뢰 금지).
- 페이지네이션: 반응형 컬럼 수에 따라 페이지당 행 수 가변(현재 데스크톱 기준 4행 단위) — **[신규]** API는 고정 `limit`(예: 20) 기반 offset/cursor 페이지네이션으로 단순화 권장.

### 3.2 부원 상세
- 표시 필드: 아바타, 상태 배지(ACTIVE/PROBATION/INACTIVE/WITHDRAWN), 화이트리스트 배지, 직책 라벨(`개발`/`회장`·`부회장`·`총무`/`운영진`/없음), 세션, 자기소개, 소속 팀+팀 내 역할, 학과, 전화번호(`phone_private` 존중), 가입일.
- 본인 프로필인 경우: "프로필 수정" 버튼 + 받은 팀 초대함(수락/거절) + 내가 보낸 팀 가입 신청함(취소).
- 운영진만 보이는 패널: 경고 이력, 상태 변경, 직책 부여 — **단, 이 패널들은 반드시 서버가 호출자 role을 재검증한 뒤 데이터를 내려줘야 한다.** **[MOCK]** 현재는 클라이언트 `ViewerSwitch`로 미리보기만 전환 가능한 취약한 방식 — **[신규]** 실서비스는 이 토글을 완전히 제거하고 실제 세션 role로만 분기.

### 3.3 내 프로필 수정 (`/profile/edit`)
- 필드: 아바타 소스, 닉네임(≤20), 자기소개(≤60), 세션(다중)+세션별 경력 연차, 선호 장르, 전화번호, 학과, 학번, 학년.
- **공개 범위 설정**(`privacy_settings`): 이름/기수/전화번호/학과/학번/학년 각각에 대해 `all`(전체 공개) / `member`(부원까지) / `admin`(운영진까지) 중 선택. 단 전화번호·학번은 `all` 옵션 자체를 제공하지 않는다(최소 `member`부터).
- 저장은 본인 `users` 레코드 UPDATE. 상태/역할/직책/화이트리스트는 이 화면에서 수정 불가(운영진 전용 화면에서만).

### 3.4 개인정보 마스킹 규칙 [신규 — 서버 강제]
| 필드 | 옵션 | 규칙 |
|---|---|---|
| name | all/member/admin | `all` 미만 요청자에게는 표시하지 않거나 이니셜 대체 |
| generation(기수) | all/member/admin | 동일 |
| phone | member/admin | `all` 옵션 없음 |
| department | all/member/admin | |
| student_id | member/admin | `all` 옵션 없음 |
| school_year | all/member/admin | |
> 요청자 자신의 레코드는 항상 전체 필드 노출(마스킹 예외).

## 4. 팀

### 4.1 팀 목록 (`/teams`)
- 탭: 활성/비활성. 비활성 탭에는 모집 중(`recruiting=true`) 배지 표시.
- 카드: 이름, 색상(hue), 팀장, 인원수, 세션 구성, 현재 합주곡(없으면 "연습곡 미정"), 모집 배지.

### 4.2 팀 상세
- 소개, 합주곡, 세션 구성 바, 멤버 목록(팀장/부팀장은 관리 권한 보유자에게 강퇴 버튼, 팀장 본인은 강퇴 불가).
- **가입 신청**: `recruiting=true && active=true`인 팀에 한해, 비소속 부원이 메시지와 함께 신청 가능. 팀장/부팀장이 수락(→ 팀원으로 편입, 기존 소속 팀이 있었다면 자동 탈퇴 처리)/거절.
- **비활성 팀 패널**(`!active`): 비팀장/비운영진에게는 안내 문구만. 팀장은 활성화 신청서(사유 ≤200자) 제출 또는 대기 중 신청 취소 가능. 운영진은 즉시 승인/거절 단축 버튼 보유.

### 4.3 팀 생성/수정
- 생성: 이름(필수, ≤50), 소개(≤200), 합주곡(다중 선택 피커) — 생성자는 자동으로 팀장 지정, `is_active=false`(활성화 신청 필요), `is_recruiting`은 생성 시 기본값 설정 가능.
- 수정 권한 분리:
  - 팀장/부팀장: 소개, 합주곡, 모집 상태(`is_recruiting`)
  - 팀장만: 부팀장 지정/해제, 팀장 위임(위임 시 기존 팀장은 즉시 `member`로 강등)
  - 운영진만: 팀명, 색상(hue), `is_active` 직접 토글
- 팀 삭제: 운영진만. 삭제 시 소속 팀원 전원 `team_id/team_role=NULL` 처리, 관련 예약/활성화 신청 이력은 보존(soft 처리 또는 팀 스냅샷 보관 — 구현 시 결정).

## 5. 타임테이블 (`/timetable`)

### 5.1 조회
- 탭: 현재 학기/방학 + 예정된 학기/방학(오름차순).
- 그리드: 요일×시간 셀, 각 셀에 해당 팀의 예약 블록(정기/단발 구분 스타일) 표시. 운영시간 밖 시간대는 비활성 셀. `calendar_events(kind='closed')` 기간은 예약 불가 오버레이.
- 예약 오픈 배너: `terms.book_open_date/time` 이전이면 "예약 준비 중 D-n" 배너 + 조회만 가능. 오픈 이후에는 정상 예약 가능.

### 5.2 예약 생성/수정 (팀장/부팀장, 자기 팀만)
- 셀 클릭 → 인접한 빈 시간대로 드래그 확장하여 시작~종료 선택.
- 예약 생성 시 팀 선택(본인이 팀장/부팀장인 팀 중), 종류(정기/단발) 선택.
- 정기 예약 수정 시 적용 범위 선택: "이번 주만" 또는 "전체"(해당 학기 전체 반복 규칙 변경).
- 다른 팀 예약은 읽기 전용(수정 버튼 비활성 + 안내 문구).
- 삭제는 확인 모달 필요, 정기예약 삭제 시 범위(이번 주/전체)에 따라 문구 분기.

### 5.3 운영진 설정(`/admin/timetable`)
- **학기/방학 관리(Terms)**: label, 시작일, 종료일, 예약 오픈 일시(선택) CRUD. 검증: 종료일≥시작일, 오픈일시는 시작일 이전이어야 하며 날짜/시각 둘 다 설정하거나 둘 다 비워야 함.
- **운영시간(HoursConfig)**: 학기중/방학중 × 평일/주말 개장·마감 시각(6~24시 범위 select).
- **행사/휴무(CalEvents)**: kind(휴무/행사), label, 기간, 종일 여부(아니면 시작·종료 시각, 종료>시작 검증), 메모.

## 6. 공지/게시판 (`/notices`)

### 6.1 공지 종류 및 태그
- `kind='admin'`(운영진 공지): 태그 `{공연, 합주실, 회계, 행사, 모집}`, 작성은 운영진만.
- `kind='user'`(부원 공지): 태그 `{모집, 자유, 행사, 후기, 질문}`, **작성은 전 부원 가능**(권한 제한 없음).
- 목록: 고정 공지(pinned) 상단 항상 노출 + 나머지는 페이지네이션 5건/페이지.
- 상세 진입 시 조회수 +1(중복 방지 없음, 작성자 본인 열람도 카운트), `kind='admin'`인 경우 열람 기록(notice_reads) 저장 → 읽음 확인(ReadReceipt) 패널에 반영. 이 패널은 운영진 또는 작성자 본인만 조회 가능.

### 6.2 작성/수정
- 검증: 제목 필수(≤60자), 본문 필수(HTML 태그 제거 후 공백만 있으면 무효), 태그 필수.
- 수정 시 `kind`는 잠금(수정 화면에서 admin↔user 전환 불가). 신규 작성 시 admin→user로 바꾸면 `pinned`는 강제 false.
- 이미지 첨부: 여러 장 첨부 가능(개수/용량 제한은 **[신규]** 서버에서 정의 필요 — mock은 제한 없음, 실서비스는 업로드 용량/개수 상한 필요).
- 본문은 일반 텍스트(줄바꿈 유지) 또는 리치 HTML(`rich=true`) — **[신규]** 리치 HTML 저장/렌더링 시 XSS 방지를 위한 서버 sanitize 필수(mock은 `dangerouslySetInnerHTML`로 무검증 렌더링 중이었음, 반드시 개선).
- 수정/삭제 권한: 작성자 본인 또는 운영진.

### 6.3 댓글/대댓글 (부원 공지 전용, `kind='user'`만)
- 전 부원 댓글 작성 가능. 대댓글은 `parent_reply_id`로 무한 depth 연결 가능하나 UI는 depth 1까지만 들여쓰기(그 이상은 평면 표시).
- 삭제: 댓글/대댓글 작성자 본인만 가능(운영진의 타인 댓글 강제 삭제 기능은 **[신규]** — 필요 시 운영 정책 논의 후 추가, 현재 mock에는 없음).
- 대댓글 삭제 시 하위 자손 답글도 함께 삭제(cascade).
- 운영진 공지(`kind='admin'`)에는 댓글 UI 자체가 없음(읽음 확인만 존재).

### 6.4 상시 안내(References)
- 운영진 전용 CRUD(동방 비밀번호, 운영 수칙, 장비 사용 수칙 등). `secret=true`인 항목은 기본 마스킹(••••), "보기" 클릭 시 노출(클라이언트 상태, 서버에는 항상 평문 저장 — **[신규]** 민감도가 높다면 암호화 저장 고려).
- label ≤24자, value ≤40자.

## 7. 신고/제보 (`/report`)

### 7.1 신고 작성 (`/report/new`, 전 부원)
- 카테고리(부원 신고/합주실·장비/예약·노쇼/운영 건의·제보/기타), 제목(필수 ≤60), 상세내용(필수 ≤1000), 익명 여부(기본 true).
- 제출 시 `status='received'` 고정(작성자가 임의 상태 지정 불가), 신고자에게 접수 확인 알림 발송.
- 증빙 첨부 UI는 **[신규]** 실제 파일 업로드 구현 필요(mock은 장식용 placeholder).

### 7.2 내 신고 내역 (`/report/history`, 부원)
- 본인이 제출한 신고만 조회(타인 신고 열람 불가).

### 7.3 관리 큐 (`/report`, 운영진 전용)
- 상태별 카운트 카드, 필터/정렬(접수 순 우선, resolved/rejected는 후순위 동일 랭크).
- 상태 전이: `received`↔`reviewing`는 자유 토글(답변 불필요). `resolved`/`rejected`로의 전이는 전용 버튼을 통해서만 가능하며 답변(≤500자, 선택)을 함께 남길 수 있음.
- 답변은 신고당 1개만 저장(재답변 시 덮어씀, 스레드 아님) — 상태 변경 시 신고자에게 알림 발송("신고 상태가 변경됐어요" + 답변 내용 포함).
- **익명 처리**: `anonymous=true`인 신고는 관리 큐 화면에서 신고자 식별 정보를 표시하지 않음(자물쇠 아이콘 + "익명 접수"). 단, 신고자 본인의 "내 신고 내역"에서는 항상 본인 신고로 조회 가능해야 하므로 `reporter_id`는 DB에 항상 저장하고 **조회 API 응답에서만** 마스킹한다.

## 8. 알림 (`/notifications`)

- 유형: 공지(notice)/댓글(comment)/팀(team)/신고(report)/일정(schedule)/시스템(system), 유형별 아이콘·색상 고정.
- 벨 아이콘 배지: 미확인 개수(9개 초과 시 "9+" 표시). 드롭다운은 최근 6건만.
- 목록 화면 필터: 전체/안읽음/유형별(notice·comment·team·report·schedule — system은 필터 칩 없음, **[신규]** 일관성을 위해 system 필터도 추가 검토).
- 항목 클릭 시 읽음 처리 + `target_screen`/`target_params`로 딥링크 이동. 별도의 "읽음 처리" 액션과 "열기" 액션은 분리하지 않음(열람=읽음 처리).
- 전체 읽음 처리(markAllRead) 버튼 제공. 삭제/숨김 기능은 없음(현재 범위 밖).
- **[신규]** 알림은 반드시 `recipient_id`로 스코프되어야 함 — mock은 단일 사용자 가정이라 수신자 개념이 없었음, 실서비스에서는 발생 이벤트마다 대상 수신자를 명시적으로 계산해 INSERT.
- FCM 웹 푸시 연동은 PWA 구축 완료 후 별도 진행(운영 방침 확정 사항). 그 전까지는 인앱 알림(`notifications` 테이블)만 제공.

## 9. 운영진 연혁 (`/admin/{president,vice,treasurer,genleader}`)

- 회장/부회장/총무 역대 기록: 연도, 인물(현재 재적 부원과 연결 가능 — `member_id` nullable, 졸업생 등 비재적자는 이름/기수만 텍스트로 보관), 역할.
- CRUD 전부 가능, **[신규]** 운영진 권한 필요(mock에는 권한 체크 없음 — 반드시 서버에서 ADMIN 이상으로 제한).
- 기수별 기장: 현재 재적 중인 기수만 대상(`activeGens` = 현재 `users.gen`의 distinct 집합), 기수당 1명 지정, 드롭다운으로 부원 선택 시 이름/기수 자동 채움.

## 10. 운영진 관리자 화면 (`/admin/*`)

> **모든 `/admin/*` 화면은 서버에서 `role IN ('ADMIN','SUPER_ADMIN')` 검증이 선행되어야 한다.** 현재 mock 코드에는 이 검증이 전혀 없다(IA 명세서 상단 경고 참고) — 백엔드 구현 시 최우선 보완 대상.

### 10.1 모집 설정 (`recruit`)
- 필드: 모집 여부(on/off), 학기명, 헤드라인(≤40), 소개글(≤200), 대상 세션(다중 체크), 모집 기간(시작~종료일), 마감 시 안내문(≤200).
- 저장 시 `updated_at`/`updated_by` 자동 기록. 랜딩 페이지(`/`)는 이 설정을 실시간 반영(모집 여부·헤드라인·기간).

### 10.2 면접 슬롯 관리 (`interview`)
- 슬롯: 날짜, 시작/종료 시각, 정원, 메모. 종료 시각 > 시작 시각 검증.
- 슬롯 추가 시 "이전 슬롯 종료 시각 = 다음 슬롯 시작 시각"으로 자동 연결 제안(연속 슬롯 빠른 등록 UX).
- 삭제만 가능(수정 없음) — 삭제 시 확인 모달, 이미 지원자가 배정된 슬롯 삭제 시 **[신규]** 배정 해제 처리 필요(연쇄 처리 규칙 확정 필요: 배정된 지원자를 `new` 상태로 되돌릴지, 삭제를 막을지 정책 결정 필요).
- 날짜별로 그룹핑해서 표시.

### 10.3 지원자 관리 (`applicants`)
- 필터: 전체/서류확인(new)/면접예정(interview)/합격(pass)/불합격(fail), 각 상태별 카운트 표시.
- 액션:
  - `new` → 면접 슬롯 배정(`assignSlot`, 정원 초과 슬롯은 배정 불가) → `interview`
  - `interview` → 배정 취소(`cancelInterview`, 슬롯 정원 반환) → `new`
  - `new`/`interview` → 합/불 결정(`decide`) → `pass`/`fail`
  - `pass`/`fail` → 되돌리기(`revertDecision`) → 이전 단계로(슬롯 있었으면 `interview`, 없었으면 `new`)
  - 통지 처리(`notify`/`unnotify`): 상태와 독립적인 "합격/불합격 통지 완료" 플래그, 실제 발송 채널은 FCM 연동 전까지 인앱 알림으로 대체.

### 10.4 팀 활성화 관리 (`teams`)
- 대기 중인 활성화 신청 목록(체크박스 다중 선택 → 일괄 승인/거절) + 전체 팀 목록(개별/일괄 활성·비활성 토글).

### 10.5 부원 관리 (`members`)
- 목록: 검색(이름/닉네임/학과/세션), 필터(상태/운영진 여부/화이트리스트/경고 유무), 정렬(기수/이름/경고순), 페이지네이션(12건).
- 대량 선택 → 일괄 작업: 화이트리스트 on/off, 경고 추가, 활동정지(→INACTIVE)/복귀(→ACTIVE), 제명(→WITHDRAWN, 확인 모달 필요).
- 상세 드로어: 화이트리스트 토글, 운영진 권한(승격/강등 + 직책 부여 — 직책은 회장/부회장/총무 중 1명씩만, 재지정 시 기존 보유자는 자동 해제), 경고 이력(추가/삭제, "3회 제적 기준" 안내), 상태 관리(정지/복귀/제명, 사유 필수).
- **보호 대상**: `SUPER_ADMIN` 및 요청자 본인은 상태 변경 대상에서 제외(서버에서도 동일 검증 필수 — mock은 UI에서만 막고 있었음).

### 10.6 CSV 일괄 등록 (`migrate`)
- CSV 헤더 동의어 매핑(이름/닉네임/기수/세션/학과/전화번호/역할 등 다양한 표기 허용), 세션명 별칭 정규화.
- 행별 상태 판정: 정상(ok) / 중복(dup, 기존 `name+gen` 조합과 충돌) / 오류(error, 필수값 누락 등).
- "중복 포함" 토글로 중복 행도 강제 등록 가능.
- 등록 시 신규 `users` 레코드 생성 — `status='ACTIVE'`, `whitelist=false`, `team_id=NULL`. **[신규]** `auth_user_id`는 NULL로 시작(추후 `/join/existing` 흐름으로 본인이 연동).

### 10.7 게시판/신고 관리자 콘솔 (`notices`, `reports`)
- 각각 6절/7절의 기능을 관리자 레이아웃 내 인라인 콘솔로 노출(별도 라우트 이동 없이 섹션 내 상태 전환) — 기능 자체는 6절/7절과 동일, 접근 경로만 다름.

## 11. 알림 발송이 필요한 이벤트 목록 [신규 — 종합]

기존 메모(FCM 연동 계획)에 명시된 이벤트 + 본 조사에서 확인된 이벤트를 통합:

| 이벤트 | 수신자 | 발생 시점 |
|---|---|---|
| 가입 신청 완료 | 지원자 본인(향후 회원 전환 시) | `applications` 생성 |
| 면접 일정 확정 | 지원자 | 슬롯 배정 시 |
| 합격/불합격 | 지원자 | `decide('pass'/'fail')` |
| 계정 연동 완료 | 해당 부원 | `/join/existing` 제출 성공 |
| 공지 발송 | 대상 부원 전체(또는 필터 대상) | 공지 작성 시 |
| 신고 접수 확인 | 신고자 본인 | 신고 제출 시 |
| 신고 상태 변경 | 신고자 본인 | 운영진 상태 변경 시 |
| 팀 가입 신청/수락/거절 | 팀장·부팀장 / 신청자 | 각 액션 시 |
| 팀원 초대 | 피초대자 | 초대 발송 시 |
| 팀 활성화 승인/거절 | 팀장 | 운영진 결정 시 |

> FCM 연동은 PWA 구축 완료 후 진행(확정 사항). 그 전까지 위 이벤트는 모두 `notifications` 테이블 INSERT(인앱 알림)로만 처리.
